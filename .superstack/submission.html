03 · Project Description

200–500 words. Structured for scanability — judges spend 2–5 min per submission.
Copy → Submission form: Full Description

The problem
LLM agents are already moving real SOL — and they can be jailbroken, hallucinated into 99% slippage swaps, or tricked into draining wallets to unknown programs. There is no standard safety layer between a language model's output and an on-chain signature. Every agent developer is re-implementing ad-hoc checks, in different languages, with no shared test corpus of attacks.

The solution
SAK (Solana Agent Kernel) is a Rust middleware kernel that intercepts every LLM-generated transaction before signing. Every intent is simulated locally via LiteSVM — zero on-chain cost, no devnet burn. A rule engine then evaluates slippage, drain amount, program whitelist, compute budget, and priority fee. Blocked transactions never reach sendTransaction. Allowed transactions do, with a full audit trail.

Why Solana
This is only viable on Solana because transaction simulation is deterministic and fast enough to run synchronously in the agent loop. SAK uses three Solana-native primitives: LiteSVM for local pre-sign simulation, Yellowstone Geyser gRPC for same-slot push events (real devnet slots at ~2.9/sec), and Light Protocol ZK-compressed accounts for 100–1000× cheaper agent state. The kernel is impossible to build this way on EVM chains where simulation is expensive and slot times are slow.

Technical highlights
The kernel is structured as three independent Rust crates. sak-guardian: LiteSVM simulation + YAML-configurable rule engine, 20 integration tests covering the full evil corpus (slippage, drain, unknown program, compute bomb). sak-reflex: Yellowstone Geyser gRPC subscriber with auto-reconnect, emitting ChainEvent into an mpsc channel at real devnet cadence. sak-state: ZK-compressed agent state via Light Protocol (in-memory stub with full API, flush-to-ZK pending). The live demo uses NVIDIA NIM to generate adversarial trade intents, which are evaluated by real Rust Guardian code via an Axum server — no JavaScript duplicating logic.

What's next
Core kernel is functional and tested. Immediate next steps: mainnet deployment, full Light Protocol ZK-compression wiring for sak-state, and a hosted Guardian API so any Solana agent project can integrate safety checks without running their own Rust server. Longer term: NPM and Python client SDKs for non-Rust agents, and a community-maintained evil corpus that grows with new attack patterns.

06 · Technical Demo Description

Separate from pitch video — Colosseum requires both. This is the implementation walkthrough.
Copy → "Technical Demo" field (2–3 min walkthrough)

Stack: Rust (Axum, tokio, LiteSVM, yellowstone-grpc-client, tonic), standalone HTML + vanilla JS frontend, NVIDIA NIM (minimax-m2.7) for live agent intent generation.

Guardian (sak-guardian): Every evaluation calls guardian.evaluate_raw(account_keys, instructions, tx_meta). For the demo, this is exposed via POST /evaluate on the Axum server. The rule engine parses instruction data directly — system transfer discriminants (4-byte + 8-byte u64 LE), ComputeBudget discriminants (1-byte), program IDs. First rule match short-circuits. Rules are YAML-configurable via rules.yaml. 20 integration tests against a real LiteSVM instance confirm the evil corpus.

Reflex Engine (sak-reflex): Connects to Yellowstone Geyser gRPC on devnet (sol-devnet-yellowstone-grpc.rpcfast.com:443) with TLS native roots. Subscribes to slot updates at CommitmentLevel::Processed. Emits ChainEvent::SlotUpdate into an mpsc channel. The race-server forwards these to the WebSocket broadcast channel as {"type":"slot_update","slot":N} — the UI displays a pulsing live slot counter.

Integration: NVIDIA NIM → intent JSON → POST /evaluate → Rust evaluate_raw() → verdict JSON → UI renders BLOCKED/ALLOWED card with ⚙ Rust badge. If the server is unreachable, a JS fallback with the same rule thresholds runs instead (shown with orange JS badge).

07 · Why Solana Section

Copy → "Why Solana?" field

Three Solana-native primitives make SAK possible. LiteSVM: deterministic local simulation fast enough (<20ms) to run synchronously in any agent loop — this doesn't exist on EVM. Yellowstone Geyser: gRPC push stream for same-slot event delivery at ~2.9 slots/sec, enabling agents to react within the same block they observe. Light Protocol ZK-compressed accounts: 100–1000× cheaper state storage for agent history and open positions — necessary when agents are writing state on every transaction decision.
