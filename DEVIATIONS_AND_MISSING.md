## Deviation 1 — LiteSVM Simulation NOT Running in Guardian 🔴

### Claim vs Reality

| Source | Claims |
|--------|--------|
| `SAK.md` line 79 | "simulates the transaction locally inside LiteSVM before signing" |
| `SAK-1_BUILD_PHASES.md` line 227 | "Simulates the transaction locally inside LiteSVM" |
| `STATUS.md` line 56-57 | "Every transaction generated by an LLM passes through it before signing" |

**Actual implementation (`crates/sak-guardian/src/lib.rs`):**

```rust
pub struct Guardian {
    rules: RuleSet,    // ← NO LiteSVM field
}

impl Guardian {
    pub fn evaluate_raw(
        &self,
        account_keys: Vec<String>,
        instructions: &[(u8, &[u8])],
        meta: &TxMeta,
    ) -> Decision {
        let view = TxView { account_keys, instructions };
        evaluate(&self.rules, &view, meta)  // ← No simulation, just rule evaluation
    }
}
```

**Verdict:** ❌ LiteSVM is NOT running inside Guardian. The `Guardian` struct has no `LiteSVM` field. The `evaluate_raw()` method only checks rules against instruction metadata — it never simulates execution.

### What's Actually Happening

The `evaluator.rs` parses instruction data directly:
- System program: reads bincode discriminant to extract lamport amounts
- ComputeBudget: reads 1-byte discriminant for CU limit/price
- Slippage: reads from `TxMeta.slippage_bps` (agent-declared, not simulated)

### What Needs to Be Built

```rust
// Target: crates/sak-guardian/src/lib.rs

pub struct Guardian {
    rules: RuleSet,
    svm: LiteSVM,  // ← ADD THIS
}

impl Guardian {
    pub fn from_yaml(path: impl AsRef<Path>) -> Result<Self> {
        let content = std::fs::read_to_string(path)?;
        let rules: RuleSet = serde_yaml::from_str(&content)?;
        Ok(Self {
            rules,
            svm: LiteSVM::new(),  // ← Initialize SVM
        })
    }

    /// Simulate AND evaluate a full transaction.
    pub fn evaluate(&mut self, tx: &VersionedTransaction) -> Result<Decision> {
        let sim_result = self.svm.simulate_transaction(tx.clone());  // ← Run simulation
        match sim_result {
            Ok(sim) => {
                // Evaluate rules against simulation result (post balances, logs, etc.)
                evaluate_with_simulation(&self.rules, tx, &sim)
            }
            Err(e) => {
                // Simulation failed = reject
                Decision::Reject {
                    rule: "simulation_failed".into(),
                    reason: e.to_string(),
                }
            }
        }
    }

    /// Existing method for metadata-only evaluation (keep for compatibility).
    pub fn evaluate_raw(...) -> Decision { ... }
}
```

**Files to modify:**
- `crates/sak-guardian/src/lib.rs` — add `svm` field, add `evaluate()` method
- `crates/sak-guardian/src/evaluator.rs` — add `evaluate_with_simulation()` function

## Deviation 2 — Library Function, Not Runtime Kernel 🟡

### What Was Planned

```
Agent submits intent
→ SAK-1 fetches state from Geyser (Pillar 1)
→ SAK-1 simulates transaction (Pillar 2)
→ SAK-1 stores result to ZK state (Pillar 3)
→ SAK-1 signs and broadcasts
```

### What Exists

```
Someone calls evaluate_raw() with pre-built instructions
→ Rules checked
→ Decision returned
```

### Gap Analysis

| Component | Status | Notes |
|-----------|--------|-------|
| Agent intent input | ❌ Missing | No API to accept natural language or intent objects |
| Transaction builder | ❌ Missing | No code that converts intent → Solana transaction |
| Guardian evaluation | ✅ Exists | `evaluate_raw()` works, but needs LiteSVM (see Dev 1) |
| Signing | ❌ Missing | No keypair management or transaction signing |
| Broadcasting | ❌ Missing | No RPC client to send signed tx to Solana |
| ZK state storage | ❌ Missing | Pillar 3 not started |
| Geyser state fetch | ❌ Missing | Pillar 1 not started |

### For Hackathon Demo

**This is fine.** You don't need a full runtime for the hackathon. But be honest in your pitch:

> "Today we're showing Pillar 2 — the Guardian rule engine that validates every transaction before signing. The full runtime integrating all three pillars is on our roadmap."

**Don't claim:** "We are a complete agent runtime kernel" (you're not there yet).

### Component A — Transaction Generator Script

**File to create:** `demo/tx-generator/src/main.rs`

```rust
use sak_guardian::Guardian;
use sak_core::TxMeta;
use solana_sdk::transaction::Transaction;
use tokio::time::{interval, Duration};

/// Generates a mix of malicious and valid transactions
/// and feeds them to the Guardian continuously.
#[tokio::main]
async fn main() {
    let mut guardian = Guardian::from_yaml("rules.yaml").unwrap();
    let mut ticker = interval(Duration::from_secs(2));
    let mut tx_factory = TxFactory::new();

    loop {
        ticker.tick().await;

        // Random mix: 70% evil, 30% valid
        let (tx, meta) = if rand::random::<f64>() < 0.7 {
            tx_factory.random_evil_tx()
        } else {
            tx_factory.random_valid_tx()
        };

        let decision = guardian.evaluate(&tx);

        // Print for now — will pipe to WebSocket later
        println!("{}", serde_json::to_string(&decision).unwrap());
    }
}
```

### Component B — WebSocket Server

**File to create:** `demo/race-server/src/main.rs`

```rust
use axum::{
    extract::ws::{WebSocket, WebSocketUpgrade},
    response::IntoResponse,
    routing::get,
    Router,
};
use tokio::sync::broadcast;

#[tokio::main]
async fn main() {
    let (tx, _) = broadcast::channel::<String>(100);

    // Spawn transaction generator loop
    let tx_clone = tx.clone();
    tokio::spawn(async move {
        run_guardian_loop(tx_clone).await;
    });

    let app = Router::new()
        .route("/ws", get(move |ws: WebSocketUpgrade| {
            let rx = tx.subscribe();
            async move { ws.on_upgrade(|socket| handle_ws(socket, rx)) }
        }));

    println!("WebSocket server running on ws://localhost:3001");
    axum::serve(
        tokio::net::TcpListener::bind("0.0.0.0:3001").await.unwrap(),
        app,
    ).await.unwrap();
}
```
