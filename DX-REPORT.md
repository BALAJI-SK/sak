# SAK × Jupiter — Developer Experience Report

## How SAK Integrates Jupiter

SAK (Solana Agent Kernel) is a Rust execution safety kernel for Solana AI agents. It integrates Jupiter at the **kernel level** — not as an app that calls Jupiter, but as the safety layer that protects every Jupiter swap an AI agent attempts.

**The integration:**
- Jupiter V6 (`JUP6LkbZbjS1jKKwapdHNy74zcZ3tLUZoi5QNyVTaV4`) and V4 (`JUP4Fb2cqiRUcaTHdrPC8h2gNsA2ETXiPDD33WcGuJB`) are whitelisted in SAK's Guardian program list
- Every Jupiter swap generated by an AI agent is simulated in LiteSVM and evaluated against 2,010 safety rules **before** the agent signs
- Slippage caps (200 bps default) are enforced on Jupiter transactions — if an agent tries a 99% slippage swap through Jupiter, SAK blocks it
- The demo dashboard uses Jupiter as the primary swap protocol in all attack/valid scenarios
- The NVIDIA AI agent prompts specifically instruct the LLM to use Jupiter V6 for swaps

**What this means for Jupiter:** Any AI agent framework that integrates SAK automatically gets safe Jupiter swaps. SAK sits underneath — the agent doesn't need to know about slippage protection, drain detection, or compute bomb prevention. SAK handles it.

## Onboarding Experience

**Time from landing on developers.jup.ag to understanding what's available:** ~10 minutes.

The platform layout is clean. Having API keys, docs, and analytics in one place is a significant improvement over the old setup where you'd bounce between multiple domains. The single API key model is the right call.

**What worked:**
- The API reference is well-structured with request/response examples
- Having all APIs (Swap, Price, Tokens, Lend, Trigger, Recurring, Prediction, Perps) under one key removes friction
- The "What you're building with" section in the bounty description is genuinely useful as a quick reference

**What took longer than it should:**
- Understanding the difference between `/order` + `/execute` (managed) vs `/build` (raw) for Swap V2 took a few reads. A simple comparison table at the top of the Swap V2 docs would help.
- The relationship between the old swap API and Swap V2 isn't immediately clear. Am I migrating? Is the old one deprecated? A one-line status indicator would resolve this.

## API Feedback

### Swap V2

We haven't yet migrated SAK's demo from direct Jupiter program calls to the Swap V2 API. The current integration is at the program-ID level (whitelisting Jupiter programs in Guardian). Integrating `/order` + `/execute` would let SAK agents actually execute safe swaps through Jupiter's routing, not just validate that the program is whitelisted.

**What we'd want:** A Rust SDK or at minimum well-documented REST endpoints with Rust/reqwest examples. The current docs show curl and JavaScript — adding Rust would help the kernel/infra developer audience.

### Price API

SAK currently proxies CoinGecko for SOL/USD pricing with a 60-second server-side cache. Jupiter's Price API would be a better fit — it's Solana-native, likely faster, and wouldn't have the CORS issues CoinGecko has on localhost.

**Migration path:** Replace `fetch_sol_price_from_coingecko()` in `race-server/main.rs` with a Jupiter Price API call. The endpoint structure looks straightforward.

### Tokens API

SAK's `tokens-blocklist.yaml` contains 2,000 long-tail SPL mints generated from `solana-labs/token-list`. Jupiter's Tokens API with verification status and organic scores would be a significant upgrade — we could block unverified tokens programmatically instead of maintaining a static blocklist.

**What we'd want:** A bulk endpoint or at least a way to check verification status for multiple mints in one call. Checking 2,000 tokens one-by-one isn't practical.

### Trigger (Limit Orders)

Interesting for agent use cases. An AI agent could set a limit order through SAK's Guardian — "only execute this swap if SOL drops to $140" — and SAK would validate the order parameters before submission. Not yet integrated but the API surface looks clean.

### Agent Skills

We installed the Jupiter Agent Skills to test. The structured context files are useful for giving coding agents Jupiter-specific guidance. However, the skills are focused on "how to call the API" — they'd be more valuable if they included "common pitfalls" and "edge cases" sections based on real developer feedback.

## AI Stack Feedback

### Agent Skills
- **What worked:** Structured context files that coding agents can consume directly. Good for onboarding.
- **What didn't:** Skills don't include error handling patterns or edge cases. "Here's how to call /order" is useful; "here's what happens when the route changes mid-execution" is more useful.
- **What's missing:** A skill specifically for AI agent safety — how to validate Jupiter swaps before signing. SAK essentially fills this gap, but Jupiter could provide a reference implementation.

### Docs MCP
- Not tested in this build. We referenced docs directly via the web.

### Jupiter CLI
- Not tested in this build. SAK's demo uses HTTP API calls, not CLI execution.

### llms.txt
- Useful for indexing. We referenced it during development to find relevant endpoints.

## What We Wish Existed

1. **Rust SDK** — Jupiter's ecosystem is heavily TypeScript. A first-party Rust SDK would help the kernel/infra developer audience that builds in Rust.

2. **Swap V2 migration guide** — A clear "if you're on the old API, here's exactly what to change" guide. Not just a changelog — a step-by-step migration.

3. **Bulk token verification endpoint** — Check 100+ mints in one call for verification status and organic scores. Essential for anyone building token allowlists/blocklists.

4. **Agent safety reference** — A Jupiter-endorsed guide on "how to safely let AI agents use Jupiter." Slippage protection, route validation, MEV awareness. SAK provides this, but it should be in Jupiter's docs too.

5. **Error response standardization** — Consistent error response format across all APIs with machine-readable error codes. When an agent gets an error, it needs to know why programmatically, not parse a human-readable message.

6. **Rate limit documentation** — Clear rate limit numbers per API key tier. We hit some ambiguity during testing.

---

## What Sucked (Honest Feedback)

1. **The old API vs new API confusion** — We spent 30 minutes figuring out if we should use the old swap endpoint or Swap V2. The docs should have a deprecation timeline at the top.

2. **No Rust examples** — Every code sample is JavaScript or curl. For a kernel-level project in Rust, we had to translate everything manually.

3. **Price API CORS** — If Jupiter's Price API has the same CORS restrictions as CoinGecko, that's a problem for browser-based demos. We proxy through our server for this reason.

4. **Token metadata endpoint** — We wanted to check if a mint is "verified" but the response includes a lot of fields we don't need. A lightweight `/is-verified` endpoint would be faster for bulk checks.

---

## Summary

SAK integrates Jupiter at the kernel level — not as a consumer, but as the safety layer that protects every Jupiter swap an AI agent attempts. The Jupiter Developer Platform's single-key model is a significant improvement. The main friction points are the lack of Rust SDK/examples and the old-vs-new API confusion.

The most impactful Jupiter feature for SAK would be the Tokens API for programmatic token verification — it would replace our static 2,000-mint blocklist with live verification data.
